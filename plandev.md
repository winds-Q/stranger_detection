# 陌生人检测报警系统后续开发计划

## 1. 计划目标

将当前项目从可运行原型提升为适合笔记本长期运行的陌生人检测系统，重点改善：

- 识别和报警准确性
- CPU、内存和磁盘占用
- 摄像头及网络异常恢复能力
- 告警事件的可追踪性
- Web 管理端安全性
- 部署、维护和测试体验

## 2. 当前基础能力

当前已经具备：

- 摄像头实时采集
- 熟人照片注册和识别
- 陌生人临时身份区分
- 同一陌生人三分钟报警冷却
- 不同陌生人分别报警
- 邮件报警及截图附件
- 本地截图保存和数量清理
- Web 启动、停止、上传和删除熟人照片
- Web 邮件配置和实时日志
- 摄像头启动失败反馈
- SMTP 465 SSL 和 STARTTLS 支持
- 基础自动化测试

## Phase 1：降低误报和资源占用

### 1.1 检测帧率控制

任务：

- 增加 `processing.detect_every_n_frames` 配置。
- 增加 `processing.max_detection_fps` 配置。
- 支持缩小图像后检测，再将坐标映射回原图。
- 非检测帧可以继续读取摄像头，但跳过人脸编码。
- 日志记录实际处理 FPS 和平均耗时。

建议默认配置：

```yaml
processing:
  detect_every_n_frames: 3
  max_detection_fps: 4
  detection_scale: 0.5
```

验收标准：

- CPU 占用相比当前版本明显下降。
- 640×480、HOG 模型下检测流程保持稳定。
- 缩放后的人脸位置映射正确。
- 不影响 Web 启动和停止检测。

### 1.2 连续多帧确认

任务：

- 陌生人首次出现时不立即报警。
- 按临时身份累计最近若干帧的出现次数。
- 达到持续时间或命中帧数后确认事件。
- 短暂误识别、画面模糊或单帧异常不触发报警。
- 增加确认参数配置。

建议默认配置：

```yaml
detection_confirmation:
  window_seconds: 3
  required_hits: 4
  minimum_duration_seconds: 1
```

验收标准：

- 单帧误识别不会报警。
- 陌生人持续出现约 1～2 秒后能够报警。
- 不同陌生人的确认计数互不影响。
- 熟人不会进入陌生人确认队列。

### 1.3 自动化测试

任务：

- 测试跳帧和 FPS 限制逻辑。
- 测试单帧陌生人不报警。
- 测试连续命中后报警。
- 测试不同陌生人的确认状态互不干扰。

## Phase 2：建立完整陌生人事件模型

### 2.1 陌生人进入、停留和离开状态

将当前单纯的冷却逻辑升级为事件状态机：

```text
候选出现
  ↓ 连续确认
已确认
  ↓
持续停留
  ↓ 超过离开判定时间未出现
已离开
  ↓ 再次出现
新事件
```

任务：

- 为每个临时身份维护首次出现、最后出现和命中次数。
- 增加离开判定时间。
- 陌生人离开后再次出现时创建新事件。
- 支持“离开后立即重新报警”，不必等待原冷却结束。
- 限制临时身份数量并清理过期数据。

建议默认配置：

```yaml
stranger_tracking:
  match_tolerance: 0.5
  leave_timeout_seconds: 30
  identity_retention_seconds: 3600
  max_identities: 200
```

验收标准：

- 同一陌生人持续停留期间不会频繁创建事件。
- 离开超过 30 秒后再次出现会创建新事件。
- 两名陌生人同时出现时分别维护事件。
- 过期身份和事件状态会自动清理。

### 2.2 改进陌生人身份匹配

任务：

- 每个陌生人保存多个近期人脸编码，而非单个平均编码。
- 结合人脸距离、位置和时间进行匹配。
- 计算相邻帧人脸框中心点距离和 IoU（交并比）。
- 结合移动方向与速度，降低多人交叉时的临时 ID 跳变。
- 避免快速平均导致身份编码漂移。
- 为匹配结果记录距离和可信程度。
- 评估是否需要引入 OpenCV Tracker、ByteTrack 或 DeepSORT。

推荐实施顺序：

1. 多编码样本匹配。
2. 人脸框中心点和移动距离匹配。
3. 多人交叉场景测试。
4. 必要时再接入专业跟踪算法。

验收标准：

- 同一人轻微转头、戴眼镜或光照改变时尽量保持相同 ID。
- 不同人员不应被轻易合并。
- 多人同时出现时 ID 跳变明显减少。

### 2.3 报警截图增强

任务：

- 在完整画面中绘制陌生人红框和临时 ID。
- 熟人使用绿色框并显示姓名。
- 保存完整画面和陌生人人脸裁剪图。
- 邮件中包含临时身份、检测时间和事件编号。
- 一帧多人时准确标注对应身份。

验收标准：

- 邮件截图可直观看出被报警的人。
- 截图中的框和实际人脸位置一致。
- 同一帧多人的标签不会错位。

## Phase 3：提升摄像头和通知稳定性

### 3.1 摄像头自动重连

任务：

- 统计连续取帧失败次数。
- 达到阈值后释放摄像头。
- 间隔数秒重新打开摄像头。
- 支持配置重试次数和重试间隔。
- Web 状态显示“正常、断开、重连中、失败”。
- 支持扫描可用摄像头设备编号。

建议默认配置：

```yaml
camera:
  device_id: 0
  width: 640
  height: 480
  read_failure_threshold: 10
  reconnect_interval_seconds: 5
```

验收标准：

- 临时拔出或占用摄像头后程序不会崩溃。
- 摄像头恢复后程序能够继续检测。
- 无法恢复时 Web 页面显示明确错误。

### 3.2 邮件异步发送

任务：

- 将邮件发送放入独立工作线程或任务队列。
- 检测线程只负责生成告警任务。
- 设置队列最大长度，防止网络异常导致无限积压。
- 增加失败重试和指数退避。
- 记录邮件发送状态和失败原因。

验收标准：

- SMTP 请求不会阻塞摄像头检测。
- 邮件服务器短暂失败后能够自动重试。
- 队列满时有明确日志且程序不会失控。

### 3.3 健康状态接口

扩展 `/api/status`：

```json
{
  "running": true,
  "camera_state": "connected",
  "known_faces": 5,
  "last_frame_time": "2026-07-16T15:00:00",
  "last_detection_time": "2026-07-16T15:00:00",
  "last_alert_time": "2026-07-16T14:58:00",
  "alerts_today": 3,
  "smtp_configured": true,
  "detection_fps": 3.8,
  "camera_reconnecting": false,
  "alert_queue_pending": 0
}
```

验收标准：

- Web 页面可以区分“线程存活”和“系统正常工作”。
- 摄像头、SMTP、熟人库及最后活动时间均可查看。
- 可以看到摄像头重连状态、报警队列积压数量和当日报警次数。
- 状态接口不得返回 SMTP 授权码或其他敏感信息。

## Phase 5：安全性和隐私保护

### 5.1 Web 登录认证

任务：

- 增加管理员登录页面。
- 使用安全密码哈希，不保存明文密码。
- 使用 Session 认证。
- 对启动、停止、配置、上传、删除接口进行权限校验。

验收标准：

- 未登录用户不能访问管理接口。
- 密码不会以明文形式写入日志或配置。
- Session 过期后需要重新登录。

### 5.2 CSRF 和接口安全

任务：

- 为修改类请求增加 CSRF 防护。
- 限制上传大小和类型。
- 保持文件名净化和路径校验。
- 为危险操作增加二次确认。
- 增加基础请求限速。

### 5.3 敏感配置管理

任务：

- 新增 `config.example.yaml`。
- 将真实 `config.yaml` 加入 `.gitignore`。
- SMTP 授权码不再通过 Web 写入 YAML。
- Windows 优先使用环境变量或凭据管理器。
- 对日志进行敏感信息过滤。
- Web 只显示“SMTP 已配置/未配置”，不回填或返回授权码掩码。
- 增加“发送测试邮件”操作，只返回成功或脱敏后的失败原因。
- 测试邮件接口增加认证、CSRF 和请求限速。

验收标准：

- 仓库中不包含真实邮箱授权码。
- API 不返回真实授权码。
- 日志不会出现密码或完整认证信息。
- 测试邮件过程中不会在响应或日志中泄露授权码。

## Phase 6：熟人注册和实时预览

### 6.1 Web 摄像头预览

任务：

- 提供低帧率 MJPEG 或 WebSocket 预览。
- 显示熟人绿色框和陌生人红色框。
- 显示姓名、临时身份和检测 FPS。
- 预览使用经过处理的缓存帧，避免重复运行检测。
- 可在配置中关闭预览。

验收标准：

- 开启预览后 CPU 占用增幅可控。
- 多浏览器连接不会重复开启摄像头。
- 停止检测后预览自动停止或显示占位状态。

### 6.2 熟人注册向导

任务：

- 用户输入熟人姓名。
- 从摄像头连续采集 5～10 张照片。
- 检查每张照片只包含一张人脸。
- 检测照片清晰度、亮度和姿态差异。
- 自动生成安全文件名和人员目录。
- 注册完成后自动重新加载熟人库。

建议目录结构：

```text
known_faces/
├── alice/
│   ├── 001.jpg
│   ├── 002.jpg
│   └── 003.jpg
└── bob/
    ├── 001.jpg
    └── 002.jpg
```

验收标准：

- 无人脸、多人脸和严重模糊照片会被拒绝。
- 同一人员可以保存多个有效样本。
- Web 页面能按人员而非按文件展示熟人列表。

### 6.3 熟人图片即时质量校验

任务：

- 在普通上传接口中立即检查是否检测到人脸。
- 拒绝无人脸或包含多张人脸的照片。
- 检查人脸区域尺寸，拒绝人脸过小的照片。
- 使用拉普拉斯方差检查模糊程度。
- 检查过曝、欠曝和对比度过低。
- 检查是否与现有熟人样本高度重复。
- 返回明确且不包含本地绝对路径的失败原因。

建议配置：

```yaml
face_upload_validation:
  min_face_size: 80
  min_blur_score: 80
  min_brightness: 35
  max_brightness: 225
  require_single_face: true
```

验收标准：

- 无人脸、多人脸、严重模糊和曝光异常照片无法进入熟人库。
- 合格照片上传后能够自动重新加载熟人库。
- 校验失败不会覆盖原有同名照片。

### 6.4 Web 配置中心

任务：

- 管理摄像头编号、分辨率和自动重连参数。
- 管理检测 FPS、跳帧数量和缩放比例。
- 管理熟人、陌生人匹配阈值和多帧确认参数。
- 管理离开判定、截图保留、邮件队列和重试参数。
- 每项配置显示用途、推荐范围和风险提示。
- 保存前进行类型、上下限和组合关系校验。
- 说明哪些配置需要重启检测后生效。

验收标准：

- 无效参数无法保存，并显示具体字段错误。
- 修改非敏感配置不会覆盖已有但未展示的敏感配置。
- 保存过程中不在前端或日志中输出授权码。

### 6.5 摄像头设备扫描和测试

任务：

- 增加“扫描摄像头”接口和按钮。
- 列出可用设备编号、实际分辨率和打开状态。
- 支持短时测试画面或单帧预览。
- 测试结束后立即释放摄像头，避免占用检测任务设备。
- 检测正在运行时禁止并发扫描同一设备。
- 保存用户选中的设备编号。

验收标准：

- 可以在 Web 页面选择实际可用的摄像头。
- 扫描失败不会导致 Web 服务或检测线程退出。
- 扫描和测试不会遗留摄像头句柄。

### 8.4 性能和稳定性测试

测试场景：

- 连续运行 24 小时。
- 连续运行 7 天。
- 单人持续停留。
- 多人同时进入和离开。
- 熟人和陌生人同时出现。
- 摄像头断开后恢复。
- SMTP 无网络后恢复。
- 磁盘空间不足。
- Web 重复启动和停止检测。

验收指标建议：

- 检测线程无未处理异常。
- 内存占用无持续增长。
- 身份缓存和告警队列数量有明确上限。
- 摄像头恢复后可自动继续检测。
- SMTP 异常不阻塞检测。
- 误报率和漏报率有测试记录。
